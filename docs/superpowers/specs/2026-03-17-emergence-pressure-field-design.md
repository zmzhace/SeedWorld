# Emergence Pressure Field Design

Date: 2026-03-17
Topic: Improve emergent behavior by strengthening situation pressure rather than adding behavior rules.

## 1. Goal

Upgrade the current system from "multiple mechanisms existing side by side" into "the world forms a pressure field first, then agents act naturally from within it."

The focus is not to add more behavior types, triggers, or explicit strategic instructions. The focus is to:

- let the world setting determine what matters most right now
- let resources, status, environment, and information converge into a shared pressure field
- let each agent feel that it is inside a distribution order, threat structure, and opportunity window
- let each tick leave structural consequences that shape the next tick

## 2. Core Principles

### 2.1 Describe situation, not behavior
The engine should never tell the model what strategy to choose. It should only surface:

- what the agent lacks
- who controls access to it
- what order is hardening around the agent
- what the agent risks losing by doing nothing

Behavior remains fully generated by the LLM.

### 2.2 Let world setting decide pressure weights
Resources, status, environment, and information all exist, but their relative importance must be determined by `world.environment.description` and current world state, not by fixed engine policy.

### 2.3 Turn systems into a causal network
Reputation, resources, tension, and memes should not feel like separate dashboards. They should reinforce each other:

- resource control changes status
- status changes distribution power
- information changes legitimacy and alignment
- tension changes whether friction stays latent or explodes

### 2.4 Keep prompts subjective
Agents should receive felt reality, not system reports. Prompt text should be shaped as lived pressure:

- you are being pushed to the back of the line
- X controls the path to what you need
- silence will be read as acceptance of the current order

### 2.5 Persist structure across ticks
Each tick should leave durable consequences such as:

- who now depends on whom
- who can allocate and gatekeep
- who is being marginalized
- which order is stabilizing or breaking down

## 3. Non-Goals

This design does not aim to:

- add many specialized behavior systems
- add theme-specific rules for betrayal, alliance, rebellion, etc.
- make the story director more prescriptive
- turn agents into fixed archetype behavior trees

## 4. Proposed Architecture

### 4.1 World Pressure Profile
A world-level abstraction layer that compiles world setting plus current global state into the dominant pressures shaping the world.

#### 4.1.1 Compilation contract
The profile must be built once per tick before agent execution. All first-wave agents in the same tick read the same profile snapshot. If wave execution changes world state materially, second-wave agents may read a refreshed profile derived from the updated intermediate state.

This compilation should be deterministic code, not a separate LLM pass. The project goal is to make the world state itself rich enough to drive emergence, so pressure interpretation should stay inspectable and testable.

Allowed evidence sources:

- `world.environment.description`
- persisted systems state (`reputation`, `resources`, `tension`, `memes`, `knowledge_graph`, `attention`, `collective_memory`)
- active narratives and recent events
- location and population distribution

`world.environment.description` influences prioritization by contributing abstract world traits rather than theme-specific rules. The compiler should extract only high-level world biases such as:

- scarcity-normalized vs abundance-normalized world
- status-rigid vs status-fluid world
- access-open vs access-gated world
- violence-legitimate vs violence-costly world
- knowledge-public vs knowledge-controlled world

These biases are not behavior instructions. They are modifiers on how current world evidence is interpreted.

#### 4.1.2 Minimum data model
The minimum output schema should be an internal typed object equivalent to:

- `dominantPressures`: ranked list of pressure axes with normalized weight and evidence
- `powerBasis`: ranked list of what currently converts most reliably into leverage
- `distributionPattern`: current pattern such as open contest, gatekeeping, patronage, or hidden control
- `legitimacyBasis`: what kinds of claims are socially accepted in this world right now
- `faultLines`: ranked fractures with affected groups, places, or relationship clusters
- `volatileZones`: ranked locations or relationship axes likely to intensify
- `evidenceTrace`: lightweight references to supporting systems and events

Each ranked item should include:

- `kind`
- `weight`
- `summary`
- `evidence`

This profile may be persisted under a lightweight per-tick snapshot for debugging and replay, but it is not a new long-lived mechanism system.

#### 4.1.3 Why this exists
This layer answers:

- what is most worth fighting over in this world right now?
- why does controlling it create leverage?
- who is structurally advantaged or exposed?

This is the mechanism that lets world setting decide whether scarcity, status, environment, or information leads.

### 4.2 Situation Compiler
An agent-level layer that turns global system state into immediate, subjective situation pressure.

Inputs:

- reputation
- resources
- tension
- memes
- location and co-location
- recent events
- knowledge graph
- world pressure profile

#### 4.2.1 Minimum data model
The minimum internal output schema should include ranked situation items for:

- `needsPressure`
- `dependencyPressure`
- `statusPressure`
- `obstruction`
- `leverage`
- `exposure`
- `opportunityWindow`
- `inactionCost`
- `evidenceTrace`

Each item should include:

- `summary`
- `severity` or `urgency`
- `sourceEntities`
- `sourceLocations`
- `relatedPressureAxis`
- `evidence`

The compiler must limit payload by selecting only the top few high-coherence items, not dumping every possible pressure into the prompt.

#### 4.2.2 Prompt-compilation guardrails
The compiler must remain a situation renderer, not a soft director.

Allowed phrasing:

- observable asymmetries
- blocked or privileged access
- contested legitimacy
- dependencies already grounded in state
- likely losses from inaction grounded in the current order

Forbidden phrasing:

- imperative language (`you should`, `you must`, `best move`)
- evaluative tactical advice (`smartest path`, `obvious response`)
- single-path framing that implies one privileged action
- invented consequences not grounded in state or profile evidence

Implementation rule: whenever the compiler emits a pressure, it must preserve open affordances. It may say the agent is exposed, blocked, dependent, or losing standing, but it must not imply whether the agent should appease, challenge, evade, hoard, negotiate, or wait.

Acceptable examples:

- `X now controls the faster route to the well, and others are already benefiting from that access.`
- `Recent distribution has left you closer to the back of the queue than before.`
- `Your standing makes your claims easier to dismiss in this location.`

Unacceptable examples:

- `You should confront X before the queue hardens.`
- `The obvious move is to align with Y.`
- `You need to seize control now.`

#### 4.2.3 Why this exists
The purpose is to upgrade the prompt from "many facts are present" to "a coherent situation is pressing on you now."

### 4.3 Cross-System Consequence Layer
A lightweight persistence layer that turns key actions in the current tick into structural consequences for later ticks.

It should capture only the highest-value situation shifts, such as:

- changes in access control
- dependency formation
- legitimacy gain or erosion
- hotspot intensification
- normalization of exclusion or favoritism

This layer does not direct the story. It only makes already-emerged dynamics durable.

#### 4.3.1 Ownership rules
To avoid semantic duplication, storage ownership must be explicit:

- `resources` owns scarcity, possession, access-control, and distribution consequences
- `reputation` owns standing, influence, deference, and dismissibility consequences
- `tension` owns escalation readiness and hotspot instability
- `memes` owns narrative spread, norm drift, and expectation contagion
- `knowledge_graph` owns structural relations such as `depends_on`, `controls_access_to`, `blocked_by`, `aligned_with`, `challenged_by`

A lightweight `situation_snapshot` may exist only as a derived read model for prompt compilation and debugging. It is never canonical if the same concept already belongs to an existing system.

Prompt compilation reads concepts back from their canonical owners first, then optionally uses `situation_snapshot` as a cached summary layer.

#### 4.3.2 Merge and conflict rules
If multiple systems imply the same shift:

- structural relation goes to `knowledge_graph`
- material control goes to `resources`
- social standing goes to `reputation`
- emotional or explosive readiness goes to `tension`
- shared interpretation or belief goes to `memes`

The snapshot layer may summarize these shifts, but never redefine them.

## 5. Mapping to Existing Files

### 5.1 `src/server/llm/agent-decision-llm.ts`
This is the primary prompt-side integration point.

Current strengths:

- colocated and distant people are already separated
- relationships, memories, rumors, recent events, and active narratives are already present
- pressure language already exists in limited form

Current limitation:

- the prompt mostly lists information instead of compiling it into a lived situation

Recommended changes:

- add `buildWorldPressureProfile(world)`
- add `compileAgentSituation(agent, world, profile)`
- place compiled situation pressure before broader world context in the prompt

The prompt should answer:

- what matters now
- why it matters to this agent
- who benefits if the agent does nothing

It must still avoid telling the agent what action to take.

### 5.2 `src/engine/orchestrator.ts`
This is the main tick-to-tick continuity integration point.

Current strengths:

- systems are already hydrated from `world.systems`
- snapshots are already exported and persisted across ticks

Current limitation:

- persistence exists, but not enough world-structure semantics are being persisted

Recommended changes:

- build the world pressure profile once near tick start after hydration
- let all agents in the same wave share that profile snapshot
- after first-wave actions update world state, allow second-wave agents to consume a refreshed profile if intermediate state materially changes access, standing, or hotspot structure
- add a post-decision structural consequence pass after tick execution
- extract high-value shifts in access, dependency, legitimacy, and hotspots
- write them back into canonical existing systems, with optional derived `situation_snapshot` generation

The orchestrator should answer:

- what order is forming?
- who benefits from that order?
- who is constrained by it?

It should not become a director that scripts future events.

### 5.3 Existing systems as pressure sources
Use existing systems more as sources of world pressure instead of isolated readouts.

- reputation system: who can influence allocation, who is believed, who is ignored
- resource competition system: who controls inputs, access points, and scarcity
- dramatic tension system: which frictions are near explosive thresholds
- meme propagation system: which narratives are shifting legitimacy and coordination
- knowledge graph: durable structure for dependency, blockage, alignment, challenge, and access control

## 6. Prompt Shape

The current philosophy in `CLAUDE.md` should be preserved: desire first, then immediate context and pressure.

Recommended prompt order:

1. desire / what the agent most wants to secure or obtain
2. immediate situation pressure
   - what the agent lacks
   - who controls it
   - who currently benefits from the order
   - what doing nothing will cost
3. scene
4. identity and social position
5. nearby or relevant people
6. memory
7. broader world context

Prompt wording rules:

- describe situation, never prescribe strategy
- emphasize cost and leverage, never issue commands
- keep future possibilities open instead of forcing narrative conclusions
- every pressure statement must be traceable to existing state, profile evidence, or immediate scene context

## 7. Data Flow

Recommended decision-time flow:

`world state`
→ `hydrate systems`
→ `build world pressure profile (once per wave snapshot)`
→ `compile agent situation`
→ `prompt sections ordered by felt urgency`
→ `LLM decision`
→ `system_feedback`
→ `structural consequences`
→ `write canonical systems`
→ `optional derived situation snapshot`
→ `persisted world state`

This extends the existing loop without replacing it.

## 8. Runtime Semantics

To keep agents within the same tick aligned on the same world situation:

- the world pressure profile is built after system hydration and before agent execution
- all agents within a single execution wave read the same profile snapshot
- first-wave actions may alter the intermediate world state through existing execution flow
- before the second wave, the profile may be recomputed if first-wave changes materially affect access, legitimacy, scarcity, or hotspot intensity
- `compileAgentSituation` runs per agent against the current wave snapshot so the subjective situation stays local while the world pressure field stays shared

This gives same-wave consistency without preventing later agents from reacting to meaningful world shifts.

## 9. Validation Strategy

Success should be measured by situation formation, not by surface-level activity.

### 9.1 Structural signals
Look for:

- concentration of resource control in agents or locations
- growth of dependency edges
- reputation affecting access
- stable hotspots instead of random drift

### 9.2 Behavioral diversity signals
Look for:

- multiple responses to similar pressure
- indirect actions such as testing, appeasing, bypassing, inciting, delaying, or leveraging others
- actions that are world-consistent even when they are not purely resource-optimal
- diversity across agents and across repeated runs under similar pressure, rather than convergence toward one repeated tactic

### 9.3 Persistence signals
Look for:

- a slight or humiliation affecting later ticks
- gradual hardening or breakdown of social order
- small events accumulating into large situations

### 9.4 World-difference signals
Use the same abstract mechanics in different world descriptions and verify that:

- the conflict focus changes
- the legitimacy basis changes
- the style of competition changes
- what counts as leverage changes materially, not just cosmetically

If all worlds still behave similarly, the design remains too template-like.

### 9.5 Philosophy-adherence checks
Add explicit audits so success cannot be faked by hidden tactic framing:

- prompt audit: verify compiled pressure text contains no commands, preferred tactics, or repeated tactic templates
- before/after seed comparison: run identical seeds and world setups before and after the compiler changes to isolate whether diversity and persistence improved
- multi-run variance check: similar starting pressure should still produce varied responses across runs and across agents
- world-bias check: changes in `world.environment.description` should alter leverage and legitimacy structure, not merely rename the same conflict loop

## 10. Rollout Plan

### Phase 1: Prompt-side situation compiler
Implement only:

- deterministic world pressure profile builder
- agent situation compiler
- prompt restructuring

Goal:

- verify that agents feel stronger situation pressure without adding behavior rules

### Phase 2: Orchestrator consequence layer
Implement:

- access changes
- dependency changes
- legitimacy shifts
- hotspot reinforcement
- exclusion or inclusion drift
- canonical write-back ownership across existing systems

Goal:

- verify that world structure accumulates across ticks

### Phase 3: Knowledge graph and emergence snapshot expansion
Only if the earlier phases work well, extend:

- graph edge vocabulary for situation structure
- emergence snapshot representation of order formation and breakdown

Goal:

- improve detection, replay, and visualization of emergent structures

## 11. Explicitly Avoid

To preserve natural emergence, avoid:

- dedicated alliance/betrayal/rebellion systems
- theme-specific hardcoded behavior rules
- over-reliance on the storyline director
- UI-first implementation before behavior validation

## 12. Recommendation

Implement the design in two practical passes:

1. first improve prompt-side world pressure compilation and agent situation compilation
2. then add structural consequence persistence in the orchestrator

Throughout both phases, avoid behavior rules and focus on expressing:

- pressure
- dependency
- access
- legitimacy
- exclusion
- opportunity windows

That is the most direct path to stronger emergence without sacrificing natural behavior generation.
